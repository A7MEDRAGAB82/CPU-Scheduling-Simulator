# Contributing to CPU Scheduling Simulator

Thank you for your interest in contributing to this project! This document provides guidelines and instructions for contributing.

## How to Contribute

### Reporting Bugs

If you find a bug, please create an issue with:
- Clear description of the bug
- Steps to reproduce
- Expected vs actual behavior
- Your environment (OS, compiler version)

### Suggesting Enhancements

We welcome suggestions for:
- New scheduling algorithms
- Performance improvements
- UI/UX enhancements
- Documentation improvements

### Code Contributions

1. **Fork the repository**
2. **Create a feature branch** (`git checkout -b feature/amazing-feature`)
3. **Make your changes** following the coding standards below
4. **Test your changes** thoroughly
5. **Commit your changes** with clear commit messages
6. **Push to your branch** (`git push origin feature/amazing-feature`)
7. **Open a Pull Request**

## Coding Standards

### Code Style

- Use meaningful variable and function names
- Add comments for complex logic
- Follow existing code formatting
- Keep functions focused and small
- Use const references when appropriate

### Documentation

- Add Doxygen-style comments for functions
- Document algorithm complexity
- Update README.md for new features

### Testing

- Test with various input scenarios
- Verify edge cases (empty input, single process, etc.)
- Ensure results match expected algorithm behavior

## Areas for Contribution

- Additional scheduling algorithms (Multilevel Feedback Queue, etc.)
- Graphical visualization of scheduling timeline
- Export results to CSV/JSON
- Unit tests
- Performance benchmarks
- Better error handling
- Input validation improvements

## Questions?

Feel free to open an issue for any questions or clarifications.
